# Q2: Business Rules, Validation Logic, and Data Transformation

## A) Implementation of Business Rules

Business rules act as the governing laws of the **Nexus Intelligent Chatbot System**, determining how operations are handled to ensure security and proper operational flow.
- **Access Control & Permissions:** Only users with an `Administrator` role can register external scripts, review audit logs, and trigger high-risk write-actions. These rules are enforced within the Authentication Service before a request hits the task execution modules. 
- **Confidence Thresholding:** The NLP Engine calculates a confidence score when extracting intents from user commands. If the score falls below a set threshold (e.g., 50%), a strict business rule dictates that the system must provide a "Fallback Response" (asking for clarification) rather than guessing and potentially executing a harmful action.
- **Sensitive Output Masking:** The BLL implements a rule stating that any logs or data returning from the Infrastructure/Script Engine that matches sensitive patterns (like passwords, keys, or internal IP structures) must be masked before being formatted into the final API response.

## B) Validation Logic

Validation logic is pivotal to our chatbot to ensure that incoming data is safe, structured correctly, and contextually valid before the backend begins processing.
- **Authentication Validation:** In the `auth/` module (e.g., `check_db.py`, `models.py`), incoming login and registration requests are validated. The software checks if email formats are correct, if passwords meet complexity guidelines, and if a username is already taken.
- **Token Checks:** Upon receiving a request from the client, the backend logic validates the JWT signature, ensuring the token is not expired and has not been tampered with.
- **Input Sanitization:** The NLP parser expects strings within a specific length to avoid payload bloat or injection attacks. The Chat API validates the schema of incoming JSON payloads using Pydantic (or similar schema validators) to ensure required fields like `user_id` and `prompt_text` are present and properly typed.

## C) Data Transformation

Data coming from the raw Data Access Layer must be heavily transformed before it's usable by the React-based Presentation Layer.
- **ORM to JSON Objects:** Our system uses a database to store user profiles, execution logs, and script metadata. When an administrator requests an audit log, the backend fetches ORM objects (e.g., SQLAlchemy/SQLModel entities). The BLL transforms these complex relational objects into flat, serialized JSON dictionaries that the frontend can easily iterate over and display in a table.
- **Intent Flattening:** The backend NLP models return highly nested probability tensors and entity trees. The BLL transforms this nested structure into a simplified intent map containing only the `action` and an array of `parameters`, which the user interface can use to clearly display what action the bot decided to take.
- **Date & Time Formatting:** Raw database timestamps are stored in UTC format. The Chat API transforms these standard timestamps into ISO 8601 strings, which the client-side UI parses and further transforms into localized, human-readable representations for chat bubbles (e.g., "Today at 2:30 PM").
